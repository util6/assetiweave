# ADR-002: Separate portable, resolved, and display paths

## Status

Accepted

## Date

2026-07-17

## Context

AssetIWeave is developed primarily on macOS but manages configuration and files
that must also work on Windows and Linux. Persisting paths such as
`/Users/alice/...`, `~/Library/Application Support/...`, or
`C:\Users\Alice\AppData\...` couples a record to the machine and operating
system that created it. Letting frontend components abbreviate arbitrary
absolute paths also makes `~` display inconsistent and can expose usernames.

Filesystem behavior is not identical across hosts. In particular, Windows path
comparison is case-insensitive, directory symlinks must be removed with
`remove_dir`, and symlink creation may require Developer Mode or elevated
permissions.

## Decision

Use three distinct path meanings throughout the application:

- **Stored path**: durable configuration uses portable text. Home paths use
  `~`; host-owned directories use `@config`, `@local-data`, `@data`, or
  `@cache`. Stored separators are `/` on every platform.
- **Resolved path**: absolute `PathBuf` values are created only at an I/O
  boundary, through `HostPathResolver` or `path_utils::expand_path`.
- **Display path**: DTOs expose an explicit `display_*` field generated by the
  backend. Runtime absolute fields remain available for Reveal and filesystem
  operations, but normal UI text uses the display field.

Concrete ownership rules:

- `backend/host_paths.rs` parses anchors, normalizes storage input, resolves
  paths for the current host, and produces display text.
- `backend/app_paths.rs` owns default paths for supported applications.
- `backend/host_filesystem.rs` owns platform-specific path comparison,
  containment, symlink creation/removal, safe path-segment validation, and
  directory-copy traversal behavior.
- Repository and settings write boundaries normalize path-bearing
  configuration before persistence. Existing Tenant records are normalized
  during initialized database startup.
- Observed runtime data such as scanned asset absolute paths and physical mount
  observations may remain absolute, because it describes the current host.
  These fields must not be used as ordinary display text.
- External adapters receive resolved absolute source and runtime paths even
  when the corresponding database or settings value is anchored.

Windows behavior is enforced by a native Windows CI job, including removal of
a broken directory symlink. A macOS cross-check is not considered proof of
Windows filesystem behavior.

## Alternatives Considered

### Persist every path as `~`

- Pros: simple and readable.
- Cons: `~/Library/Application Support` is still a macOS-specific meaning and
  cannot map to `%APPDATA%` on Windows.
- Rejected: host-owned directories require semantic anchors.

### Persist absolute paths and abbreviate them only in React

- Pros: minimal backend changes.
- Cons: records are not portable, the frontend must guess home directories,
  and every new component can accidentally expose an absolute path.
- Rejected: display formatting cannot repair incorrect persistence semantics.

### Introduce an intermediate symlink pool

- Pros: one controlled directory for links.
- Cons: adds another indirection and conflicts with the product decision to
  link target App directories directly to real source assets.
- Rejected: platform adaptation belongs in `HostFilesystem`, not another mount
  topology.

## Consequences

- New path-bearing configuration fields must declare whether they are stored,
  resolved, display, or observed runtime data.
- Code must call `expand_path` before using stored text with `std::fs`, process
  invocation, or external adapter requests.
- DTO additions generally use `display_*` fields instead of replacing runtime
  fields, preserving Reveal and Engine compatibility.
- Windows symlink failures can surface an actionable Developer Mode/elevation
  message.
- Path normalization is centralized and testable without requiring the host OS
  for pure comparison semantics; native CI remains required for actual Windows
  symlink behavior.
